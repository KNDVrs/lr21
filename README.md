Лабораторная работа №21. Архитектурный паттерн MVVM
Тема: Архитектурный паттерн MVVM (Model-View-ViewModel) в Android с Jetpack Compose.

Цели работы:
Изучить назначение и принципы архитектурного паттерна MVVM в современной Android-разработке.
Научиться разделять ответственность между логикой отображения (View) и бизнес-логикой (Model) 
через связующее звено (ViewModel).
Освоить работу с реактивным состоянием экрана с использованием StateFlow и UiState.
Интегрировать ViewModel в существующую систему навигации и внедрения зависимостей (Hilt).

Краткие теоретические сведения:
MVVM (Model-View-ViewModel) — это паттерн, обеспечивающий четкое разделение UI и бизнес-логики:
Model: Отвечает за данные (репозитории, UseCase, сетевые запросы).
View: UI-слой на Jetpack Compose. Подписывается на состояние и отображает его, не имея собственной логики.
ViewModel: Хранит состояние экрана (UiState), обрабатывает события пользователя и взаимодействует с Model. 
Благодаря интеграции с Android архитектурными компонентами, она сохраняет данные при повороте экрана.
Главное отличие от MVP: В MVVM связь между View и ViewModel реактивна. View сама наблюдает за изменениями состояния, 
что исключает необходимость прямого управления интерфейсом со стороны ViewModel.

Описание структуры MVVM в проекте:
В ходе выполнения работы была реализована следующая структура:
Определение состояния (UiState): Создан data class TasksUiState, описывающий все возможные состояния экрана (список данных, 
индикаторы загрузки isLoading, флаги добавления isAdding и ошибки).
Слой ViewModel: Реализованы TasksViewModel и TaskDetailViewModel, помеченные аннотацией @HiltViewModel. 
Они используют MutableStateFlow для управления потоком данных и viewModelScope для выполнения асинхронных операций.
Слой View (UI): Composable-функции TasksScreen и TaskDetailScreen переведены на использование collectAsStateWithLifecycle(). 
Это гарантирует, что UI обновляется только тогда, когда приложение активно.
Интеграция с DI: Зависимости (UseCase и репозитории) внедряются в конструкторы ViewModel через аннотацию @Inject.

Ключевые фрагменты кода:
Определение состояния: data class TasksUiState(val tasks: List<Task>, val isLoading: Boolean, ...)
ViewModel: Использование метода .update { it.copy(...) } для атомарного обновления состояния экрана.
Navigation: Настройка NavHost в MainActivity, где hiltViewModel() автоматически создает и предоставляет нужную ViewModel 
для каждого экрана.

Ход выполнения работы:
Описаны модели состояния (UiState) для каждого экрана приложения.
Проведена рефакторизация бизнес-логики: вызовы UseCase перенесены из UI-компонентов во ViewModel.
Реализована обработка жизненного цикла: подписка на данные во View сделана безопасной с помощью Lifecycle библиотек.
Настроена передача аргументов навигации (например, taskId) во ViewModel через SavedStateHandle.
Проведена проверка: данные корректно загружаются, отображаются индикаторы загрузки, а состояние экрана сохраняется 
при системных событиях.
Выводы
Внедрение паттерна MVVM позволило сделать UI «тонким» и декларативным. Код стал более предсказуемым, 
так как интерфейс теперь является прямым отражением состояния (UiState). 
Использование ViewModel в связке с Hilt и Jetpack Compose значительно упрощает масштабирование проекта и его тестирование, 
отделяя логику обработки данных от их визуального представления.
